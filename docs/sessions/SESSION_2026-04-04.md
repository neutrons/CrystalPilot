# Session 2026-04-04: Agent Form Control Deep Refactor

## Objective

Deep analysis of the agent-to-form-control pipeline, identifying critical
problems in how the CrystalPilot AI agent fills blanks, selects dropdowns,
clicks buttons, and interacts with the Trame/Vuetify UI â€” then implementing
fixes across 6 phases.

## Problems Identified

### Critical

1. **Single-shot tool execution** â€” The LangGraph graph was `agent -> tools -> validator -> END`. The agent could only execute ONE tool call per user message. Multi-step workflows like `set crystalsystem -> refresh_schema -> set point_group` required multiple user messages.

2. **Stale reads during agent turn** â€” `set_parameter` only updated an in-memory `config_state` dict. The actual Pydantic model write happened *after* the agent finished, so `refresh_schema` and `get_parameter` within the same turn read old values.

3. **No button/action support** â€” The agent could set fields and navigate tabs but could not trigger any UI buttons (Submit through EIC, Authenticate, Upload Strategy, Initialize Strategy, Stop Run). Users had to manually click these.

4. **Schema refresh race condition** â€” Consequence of problem #2: `refresh_schema` always read from the live model which hadn't been updated yet.

5. **Duplicate field name collisions** â€” `point_group` and `instrument` exist in both `ExperimentInfoModel` and `AnglePlanModel`. `snapshot_models` iterated sub-models in order and the last one silently overwrote the first, corrupting values.

6. **Delayed error feedback** â€” When `apply_agent_config` failed Pydantic validation, errors were stored in `_pending_bridge_errors` and only surfaced on the *next* user message. The user saw a success reply but the UI didn't change.

### Moderate

7. **Aggressive context window** â€” Only last 6 messages kept, causing the agent to forget earlier configuration in multi-turn sessions.
8. **No confirmation for destructive ops** â€” `delete_run` and `stop_run` could execute without asking the user.
9. **UI-change-during-processing race** â€” The full stale config snapshot was written back, potentially overwriting user edits made while the agent was processing.

## Implementation (6 Phases, 6 Commits)

### Phase 1: ReAct Tool Loop (`00016b8`)
**Files:** `agent.py`, `state.py`

- Replaced linear `agent -> tools -> validator -> END` with a ReAct loop: `agent -> tools -> validator -> agent` (loops back until LLM produces text or hits `MAX_TOOL_ROUNDS=6`)
- Added `tool_rounds` counter to `AgentState`
- Refactored all `_handle_*` methods to `_validate_*` methods returning note strings instead of terminal `AIMessage` states
- Validator now appends `SystemMessage` notes so the LLM sees tool results and decides next action

### Phase 2: Write-Through (`7267207`)
**Files:** `agent.py`, `bridge.py`, `chat.py` (view_models)

- Added `write_through_fn` callback to `Agent` constructor
- Added `write_single_field()` to `bridge.py` â€” writes a single field to the live Pydantic model and pushes to Trame view immediately
- Wired up in `ChatViewModel._ensure_agent`
- Called in both `_validate_set_parameter` and `_validate_set_multiple` after recording values in `config_state`

### Phase 3: Action Tools (`2bf376d`)
**Files:** `agent.py`, `tools.py`, `chat.py`, `mvvm_factory.py`, `system_prompt.md`

New tools added:
- `submit_angle_plan()` â€” wraps `MainViewModel.submit_angle_plan()`
- `authenticate_eic()` â€” wraps `MainViewModel.call_load_token()`
- `initialize_strategy()` â€” wraps `MainViewModel.reset_run()`
- `upload_strategy()` â€” wraps `MainViewModel.upload_strategy()`
- `stop_current_run()` â€” wraps `MainViewModel.stoprun()`

Architecture: `action_fns` dict passed from `ChatViewModel` (which has access to `MainViewModel`) through `Agent` to `make_tools()`. System prompt updated with multi-step chaining examples.

### Phase 4: Field Collision Fix (`892255e`)
**Files:** `bridge.py`

- Added `FIELD_OWNER` map: `{"point_group": "experimentinfo", "instrument": "experimentinfo"}`
- `snapshot_models` now uses first-write-wins with `FIELD_OWNER` override
- `write_single_field` respects `FIELD_OWNER` priority order
- Debug logging for unresolved collisions

### Phase 5: Immediate Error Feedback (`877fcaa`)
**Files:** `chat.py` (view_models)

- Bridge validation errors are now appended directly to the agent's reply text instead of being deferred to `_pending_bridge_errors`
- User immediately sees which fields were rejected and why

### Phase 6: Polish (`e0afe5c`)
**Files:** `agent.py`, `tools.py`, `chat.py`

- **Context window**: 6 -> 20 messages
- **Confirmation gates**: `delete_run` and `stop_current_run` tool descriptions now require the LLM to confirm with the user before executing
- **Compare-and-swap**: `apply_agent_config` now only writes fields that actually differ from the start-of-turn snapshot, preventing overwrites of concurrent user edits

## Test Results

All 72 existing tests pass after every phase. No test modifications needed.

## Architecture After Changes

```
User Chat Message
  |
  v
ChatViewModel.handle_submit()
  |
  +--> Pre-agent handlers (deterministic shortcuts)
  |
  +--> Agent.invoke() with current snapshot
        |
        v
     [ReAct Loop â€” up to 6 rounds]
     agent_node (LLM) --tool_calls--> tools_node (execute)
        ^                                |
        |                                v
        +-------- validator_node --------+
                  (validate, write-through to live model,
                   cross-field checks, append notes)
        |
        v
     LLM produces final text reply
  |
  v

  |
  +--> Errors? Appended to reply immediately
  |
  v

```

## Key Design Decisions

1. **Write-through via callback, not direct model access** â€” The Agent doesn't hold a reference to the Pydantic model. Instead, a `write_through_fn` closure captures the model reference. This keeps the Agent testable without requiring a full Trame stack.

2. **Action tools via `action_fns` dict** â€” Same pattern: callbacks injected at construction time. The Agent/tools layer has no dependency on `MainViewModel`.

3. **FIELD_OWNER explicit map over namespace prefixing** â€” Adding `experimentinfo.point_group` prefixes would break all existing tool calls, schema lookups, and the system prompt. An ownership map is less disruptive and extensible.

4. **LLM-level confirmation gates over code-level blocks** â€” For destructive tools, the LLM is instructed via tool descriptions to confirm first. This is more natural in conversation flow than a hard code block that would require a separate confirmation state machine.
