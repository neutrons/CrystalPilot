### Session 6 — `27a8012` — Offload figure construction to thread pool

**Files:** `view_models/main.py`

**Root cause diagnosed:** After `live_data_reduction()` completed in the thread pool, the event loop then ran `update_temporalanalysis_figure()` synchronously, which called:
1. `get_figure_intensity()` — `LinearRegression().fit()` + Plotly figure build
2. `get_figure_uncertainty()` — same
3. `update_in_view(fig)` × 3 — serializes Plotly JSON and pushes over WebSocket

This kept the event loop blocked for ~5 s after each reduction cycle, causing unresponsive tab-switching and UI interactions.

**Fix:**
- Added `_build_temporal_figures()` — builds both figures in the calling thread (intended for executor use)
- Added `async _update_figures_async(loop)` — awaits `run_in_executor(_build_temporal_figures)` to offload sklearn + Plotly work, then calls `update_in_view` with the pre-built results on the event loop
- Replaced `self.update_temporalanalysis_figure()` in `get_live_mtd_data` with `await self._update_figures_async(loop)`
- `update_temporalanalysis_figure()` kept unchanged for the startup call in `__init__` (one-time, acceptable)

**Result:** The event loop is now free during figure construction; only the lightweight `update_in_view` push remains on the event loop.
